# AI-Native 研发工程平台：端到端 Harness Engineering 设计方案

> 版本：v1.0 | 日期：2026-04-13 | 适用场景：企业级研发平台

---

## 一、核心设计理念

### 1.1 整体架构原则

基于 **Harness Engineering** 的核心范式：**Agent = Model + Harness**

平台不追求"最强模型"，而是构建一套完整的 Harness 系统，使 AI Agent 在企业级研发流程中稳定、可靠、可审计地运转。

**三大核心原则：**

- **Engineering over Intelligence**：承认模型能力边界，用工程系统补齐，而非依赖模型自身能力
- **Auditability First**：企业级合规要求每个 Agent 动作可追溯、可审查、可回滚
- **Human-in-the-Loop at Gates**：全自动执行，但在关键节点设置人工审批门禁（Gate）

### 1.2 AI 介入模式

所有阶段采用 **全自动执行 + 人工审批门禁（Approval Gate）** 模式：

```
[用户提交需求] → [AI Agent 全自动执行] → [审批门禁] → [通过后进入下一阶段]
     ↑                                                            ↓
     ←←←←←←←←←←← 驳回则返回重做 ←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←
```

### 2.2 各阶段 Agent 职责总览

## 三、Harness 六层在研发流程中的映射

### 3.1 层级定义与阶段映射

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Harness 六层架构                              │
├─────────────────────────────────────────────────────────────────────┤
│ L6 约束、校验与恢复层  │ Gate 审批规则 + 质量门禁 + 回滚策略           │
│ L5 评估与观测层        │ 各阶段质量评估指标 + 可观测性 Dashboard        │
│ L4 记忆与状态层        │ 跨阶段上下文传递 + 项目状态快照                 │
│ L3 执行编排层          │ 阶段内任务串联 + 阶段间状态机驱动              │
│ L2 工具系统层          │ 各阶段专用工具集（API 调用、CLI 操作）          │
│ L1 信息边界层          │ 角色定义 + 上下文裁剪 + 知识注入                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 各层在企业级场景的特殊增强

**L6（约束层）—— 企业级扩展：**

- 合规性约束：代码需通过安全扫描（SAST）、License 检查、代码风格检查
- 权限约束：Agent 操作范围受限于角色权限（如 Dev Agent 不可直接部署到生产）
- 审计约束：所有操作写入审计日志，不可删除
- 人工审批：Gate 节点触发审批流程，审批通过前不得进入下一阶段

**L5（评估层）—— 企业级扩展：**

- 质量指标：代码覆盖率 ≥ 80%、重复率 < 3%、圈复杂度 < 10
- 安全指标：漏洞数 = 0（阻断级）、敏感信息检测
- 业务指标：功能符合度、用户体验评分
- 审批历史：每次 Gate 的审批人、意见、时间戳

**L4（记忆层）—— 企业级扩展：**

- 项目知识库：存储 BRD、PRD、设计稿、决策记录的关联关系
- 上下文隔离：不同项目之间的上下文完全隔离
- 版本追溯：每个阶段输出物带版本号，可回溯到任意历史状态
- 交接文档：阶段间使用结构化交接文档（SOP），确保信息完整传递

## 四、各阶段详细设计

### 阶段 1：BRD 撰写

#### Agent 职责
接收产品经理（PM）或需求方的原始需求描述，自动生成规范的 BRD 文档。

#### 输入
- 原始需求（用户故事、背景描述、目标描述）
- 已有知识库（公司产品规划、历史 BRD 模板）

#### Harness 设计

**L1 信息边界层：**

- 角色设定：你是一名资深产品顾问，擅长将模糊需求转化为结构化商业文档
- 上下文裁剪：只注入与需求相关的市场信息、竞品动态、历史项目经验
- 知识注入：公司 BRD 模板格式、历史优秀 BRD 示例

**L2 工具系统层：**

- 文档生成：调用 Markdown/Word 文档生成工具
- 知识检索：RAG 检索公司历史 BRD 和市场数据
- 竞品分析：调用公开信息检索 API

**L3 执行编排层：**

```
需求理解 → 市场分析 → 竞品调研 → 商业模式画布 → 风险评估 → BRD 输出
```

**L5 评估层：**

- BRD 自检清单：目标是否清晰？市场数据是否有来源？ROI 是否可量化？
- 评估通过阈值：所有自检项通过 → 进入 Gate 审批

**L6 约束层：**

- 输出格式约束：必须包含"项目概述、市场分析、用户画像、目标与指标、风险评估"五大章节
- 长度约束：BRD 正文不超过 15 页（A4）
- Gate：PM 审批通过后方可进入 PRD 阶段

#### 输出
- BRD.md（结构化商业需求文档）
- 交接文档（含 BRD 摘要、关键决策点）

---

### 阶段 2：PRD 撰写

#### Agent 职责
接收已审批的 BRD，输出完整的产品需求文档，包括功能规格、用户故事、优先级、技术依赖。

#### 输入
- 已审批的 BRD.md
- 历史 PRD 模板
- 技术架构约束文档

#### Harness 设计

**L1 信息边界层：**

- 角色设定：你是一名资深产品架构师，擅长将商业需求拆解为可执行的技术规格
- 技术边界注入：告知当前系统架构、技术栈、已有模块能力范围

**L3 执行编排层：**

```
BRD 解析 → 功能拆解（Feature Breakdown）→ 用户故事映射 → 优先级排序（P0/P1/P2）
→ 技术依赖分析 → 非功能需求（性能/安全/合规）→ PRD 输出
```

**L5 评估层：**

- 功能完整性检查：每个 BRD 目标是否都有对应的功能点覆盖
- 排期合理性：评估各功能点的开发工作量（使用故事点）
- 一致性检查：PRD 与 BRD 目标是否一致

**L6 约束层：**

- 技术可行性约束：若某功能超出当前架构能力，需明确标注技术风险
- 合规约束：涉及用户数据的功能需标注 GDPR/数据安全合规要求
- Gate：PM 审批 + 技术负责人（TL）可行性确认

#### 输出
- PRD.md
- 功能列表（JSON 格式，供后续 Agent 消费）
- 技术风险清单

---

### 阶段 3：UX/UI 设计

#### Agent 职责
接收 PRD，输出交互方案、视觉规范，并可调用 Figma API 生成设计稿框架。

#### 输入
- PRD.md
- 设计系统规范（公司 Design Token、品牌指南）
- 已有组件库文档

#### Harness 设计

**L1 信息边界层：**

- 设计系统约束：必须遵循公司 Design Token（颜色、字体、间距、圆角）
- 平台适配约束：需要输出 Web / iOS / Android 三端适配方案
- 无障碍约束：符合 WCAG 2.1 AA 标准

**L2 工具系统层：**

- Figma API：调用 Figma 创建 Frames、Components、Auto Layout
- 设计稿导出：导出可预览链接和设计规范文档
- 设计稿标注：自动生成标注切图

**L3 执行编排层：**

```
PRD 解析 → 信息架构（IA）→ 页面流程图 → 交互方案设计
→ 视觉设计（按 Design Token）→ 设计评审材料 → Figma 输出
```

**L5 评估层：**

- Design Token 合规检查：颜色对比度、字体使用是否符合规范
- 交互一致性：同类组件在不同页面是否有一致的行为表现
- 跨端一致性：Web / iOS / Android 三端设计是否协调统一

**L6 约束层：**

- 风格约束：主色调 ≤ 2 种，字体统一（最多 2 种字体族）
- 可访问性约束：所有交互元素需有 hover/focus/active 状态
- Gate：PM 审批 + UX Designer 确认

#### 输出
- Figma 设计稿链接
- 设计规范文档（颜色、字体、间距、组件说明）
- 交接文档（页面清单、组件清单、状态说明）

---

### 阶段 4：代码开发

#### Agent 职责
接收 PRD + 设计稿，生成符合规范的完整代码实现（前后端）。

#### 输入
- PRD.md（含功能列表 JSON）
- 设计稿链接及规范
- 代码仓库地址
- 技术栈规范文档
- AGENTS.md（项目特定规则）

#### Harness 设计（最关键阶段）

**L1 信息边界层：**

```
项目 AGENTS.md 模板结构：
├── 1. 项目概述（Purpose）
│   ├── 项目背景与目标
│   └── 关键技术栈
├── 2. 架构约束（Architecture Rules）
│   ├── 目录结构规范
│   ├── 命名规范（文件/变量/函数/分支）
│   └── 模块边界规范
├── 3. 代码风格（Code Style）
│   ├── 代码格式化规则（Prettier/ESLint）
│   ├── 注释规范
│   └── 提交信息规范（Conventional Commits）
├── 4. 开发流程（Development Workflow）
│   ├── 分支管理策略（Git Flow）
│   ├── Code Review 要求
│   └── 测试覆盖率要求
├── 5. 故障案例库（Failure Cases）
│   └── 历史踩坑记录（每条包含：问题描述 + 根因 + 解决方案）
├── 6. 安全红线（Security Hard Rules）
│   ├── 禁止直接拼接 SQL
│   ├── 禁止在前端存储敏感信息
│   └── API 鉴权必须校验 token 有效期
└── 7. 合规要求（Compliance）
    ├── 数据脱敏规则
    └── 日志脱敏规范
```

**L2 工具系统层：**

- IDE 工具：代码生成、格式化、重构
- Git 操作：创建分支、提交代码、创建 PR
- CI 触发：自动触发构建流水线
- 知识库：RAG 检索内部 SDK 文档、API 文档

**L3 执行编排层：**

```
功能点拉取 → 模块拆解 → 确定依赖关系 → 编写代码（后端 → 前端）
→ 单元测试生成 → 本地自检（Lint + 类型检查）→ 提交 PR
```

**L4 记忆与状态层：**

- 任务状态文件（JSON）：追踪每个功能点的开发状态
- 中间产物管理：生成的代码文件路径、测试文件路径
- 上下文重置策略：当单次上下文利用率 > 60% 时，自动触发 Context Reset

**L5 评估层：**

- 静态检查：ESLint / Prettier / TypeScript 类型检查全部通过
- 测试覆盖率：新增代码覆盖率 ≥ 80%
- 安全扫描：SAST（静态应用安全测试）无高危漏洞
- 合规扫描：License 检查 + 敏感信息检测（如 API Key 硬编码）

**L6 约束层：**

- 单次提交范围约束：单个 PR 只做一件事（Single Responsibility）
- 分支命名约束：feat/XXX、fix/XXX、chore/XXX
- PR 描述模板：必须包含"做了什么 + 为什么这么做 + 测试方法"
- Gate：TL 审批（代码审查 + 架构审查）

#### 输出
- 代码 PR（含单元测试）
- 代码自检报告
- 安全扫描报告
- 交接文档（已实现功能清单 + 未完成功能清单 + 技术债务记录）

### 阶段 5：Code Review

#### Agent 职责
自动审查 Dev Agent 提交的代码，从质量、安全、合规、可维护性多维度给出评估报告。

#### 输入
- PR 链接 + 代码内容
- AGENTS.md
- 评审规则库（Checklist）

#### Harness 设计

**L3 执行编排层：**

```
PR 信息解析 → 代码质量扫描 → 安全漏洞扫描 → 合规检查
→ 架构合规检查 → 评审意见生成 → 评审结论（Approve / Request Changes / Reject）
```

**L5 评估层：**

- 多维度评分：代码质量（1-10）、安全性（通过/警告/阻断）、可维护性（1-10）、测试充分性（通过/需补充）
- 对比分析：与历史 PR 的代码质量趋势对比
- 问题分级：Blocker / Major / Minor / Suggestion

**L6 约束层：**

- Blocker 级别问题：必须修复，否则无法合并
- Review 通过标准：Blocker = 0，Major ≤ 2
- Gate：TL 审批确认 Agent 的评审结果（如有异议可人工覆盖）

#### 输出
- Review 报告（含问题列表、修改建议、评分）
- 自动化修复建议（Agent 可直接采纳修复）

---

### 阶段 6：自测 & 调试

#### Agent 职责
运行单元测试、集成测试，调试失败用例，修复问题直到测试通过。

#### 输入
- 代码 PR
- 测试用例规范（每个函数的输入/输出/边界条件）
- 已有测试框架配置

#### Harness 设计

**L3 执行编排层：**

```
测试发现 → 单元测试执行 → 覆盖率报告 → 失败用例分析
→ 问题定位 → 自动修复 → 重新测试 → 循环直到通过
```

**L6 约束层：**

- 覆盖率门禁：覆盖率 < 80% 阻断
- 失败重试策略：首次失败自动重试 1 次；同一测试连续失败 3 次则标记需人工介入
- 回归防护：修复时不得引入新的测试失败
- Gate：QA 确认自测报告

#### 输出
- 单元测试报告
- 覆盖率报告
- Bug 修复记录

---

### 阶段 7：联调 & 冒烟

#### Agent 职责
多模块联调、API 接口验证、冒烟测试通过。

#### 输入
- 各模块 PR
- 接口协议文档（OpenAPI/Swagger）
- 联调环境配置

#### Harness 设计

**L2 工具系统层：**

- API 测试工具：Postman / Newman
- 接口 Mock：WireMock / MockServer
- 环境管理：dev / staging 环境切换

**L3 执行编排层：**

```
环境准备 → 接口 Mock 搭建 → 按依赖顺序联调 → 冒烟测试执行
→ 问题定位 → 接口修复 → 回归验证 → 联调报告
```

**L5 评估层：**

- 接口一致性：实际接口与协议文档的符合度
- 冒烟通过率：核心流程通过率需达 100%
- 性能基线：API 响应时间 < 200ms（P95）

**L6 约束层：**

- 联调顺序约束：按依赖拓扑排序，无循环依赖
- 失败隔离：单个接口失败不得阻塞其他接口联调
- Gate：QA 冒烟测试通过确认

#### 输出
- 联调报告（含接口测试结果）
- 冒烟测试报告

---

### 阶段 8：测试用例产出

#### Agent 职责
接收 PRD，生成完整的测试用例库（手工用例 + 自动化用例）。

#### 输入
- PRD.md
- 设计稿
- 已有测试用例模板

#### Harness 设计

**L3 执行编排层：**

```
PRD 功能点提取 → 测试场景识别 → 等价类划分 → 边界值分析
→ 测试用例编写（正向/逆向/边界）→ 优先级标注（P0/P1/P2）→ 用例评审材料
```

**L5 评估层：**

- 覆盖率检查：PRD 中每个功能点是否都有对应用例
- 等价类完整性：正向/逆向/边界用例是否齐全
- 可执行性：自动化用例语法正确，可直接导入测试平台

**L6 约束层：**

- 用例格式约束：必须包含用例编号、标题、前置条件、测试步骤、预期结果
- P0 用例数约束：核心功能必须 ≥ 10 个 P0 用例
- Gate：QA Lead 审批用例库

#### 输出
- 测试用例库（可导入 TestRail / Zephyr 等平台）
- 测试计划文档

---

### 阶段 9：测试用例执行

#### Agent 职责
执行完整测试用例，录入缺陷，跟踪修复进度，进行回归验证。

#### 输入
- 测试用例库
- 最新代码版本
- 缺陷管理平台（Jira / Tapd）配置

#### Harness 设计

**L2 工具系统层：**

- 测试平台 API：自动执行用例并回写结果
- 缺陷录入：自动创建缺陷单并关联用例
- 截图工具：失败时自动截图

**L3 执行编排层：**

```
用例批次执行 → 结果记录 → 失败用例截图 → 自动提单
→ 等待修复 → 回归测试 → 缺陷关闭确认 → 测试报告
```

**L5 评估层：**

- 执行进度：已执行 / 总数、通过率
- 缺陷趋势：新增缺陷数、关闭缺陷数、遗留缺陷数
- 阻断项：是否有 P0 缺陷阻断发布

**L6 约束层：**

- P0 缺陷策略：发现 P0 缺陷立即暂停测试并通知 PM + TL
- 缺陷重开策略：回归测试失败则自动重开缺陷单
- Gate：QA Lead 确认测试通过

#### 输出
- 完整测试报告（执行结果、缺陷汇总、发布建议）

---

### 阶段 10：CI/CD 部署

#### Agent 职责
构建镜像、推送到仓库、执行 K8s 部署、灰度发布、监控验证。

#### 输入
- 测试通过报告
- 部署配置（K8s Manifest / Helm Chart）
- 环境策略（dev / staging / production）

#### Harness 设计

**L2 工具系统层：**

- Docker CLI：镜像构建和推送
- kubectl / Helm：K8s 部署
- 监控告警：Prometheus + Grafana
- 日志系统：ELK / Loki

**L3 执行编排层：**

```
代码冻结 → 触发 CI 流水线 → 构建镜像 → 安全扫描（镜像扫描）
→ 推送到镜像仓库 → Staging 部署 → 冒烟验证 → 生产灰度发布
→ 监控观察 → 全量发布（或自动回滚）
```

**L4 记忆与状态层：**

- 部署状态文件：记录当前版本、环境、发布时间
- 回滚快照：每次部署前保存上一稳定版本的快照

**L5 评估层：**

- CI 检查：构建成功 + 所有测试通过 + 安全扫描通过
- 部署后监控：P95 响应时间、错误率、CPU / 内存使用率
- 灰度指标：灰度流量占比期间错误率 ≤ 0.1%

**L6 约束层：**

- 生产部署时间窗口：仅允许工作日 10:00-17:00 部署
- 灰度策略：生产部署必须先灰度（5% → 30% → 100%），灰度期间错误率超标自动回滚
- 自动回滚触发条件：错误率 > 1% 或 P95 响应时间 > 500ms
- Gate：PM + TL + QA 三方审批

#### 输出
- 部署记录（版本、环境、时间、审批人）
- 监控报告（部署前后指标对比）
- 回滚记录（如触发）

### 5.2 上下文传递策略

- **直接传递**：下一阶段 Agent 直接读取上一阶段交接文档
- **上下文压缩**：当累计上下文 > 60% 利用率时，执行 Context Reset，保留关键摘要
- **版本锁定**：每个阶段的输出物版本锁定，后续阶段不可直接修改上游文档
